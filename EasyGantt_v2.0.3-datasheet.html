Easy Gantt v2.0.3 — Feature Datasheet

Author: Giampaolo Marucci  ·  Single-file HTML + SVG Gantt chart creator  ·  Rev. 15 Jul 2026

1 · Overview & Architecture

Easy Gantt is a self-contained, zero-dependency, single-file HTML application (≈ 4 200 lines). All scheduling logic, rendering, and user-interface code is written in vanilla JavaScript. The chart output is an inline SVG element that is regenerated on every user interaction. The application is designed to run entirely in the browser — no server, no build step.

Delivery formatSingle .html file
DependenciesNone (pure vanilla JS + SVG)
Output formatInline SVG (scalable, exportable)
Persistence formatsJSON project file, CSV export, SVG export
Responsive scalingCSS + JS transform: scale() fitted to viewport
Edit surfacePlain-text CSV textarea + tabbed option panels + inline SVG form fields

2 · Task Input Format

Tasks are entered as comma-separated or tab-separated rows in a textarea. The separator is auto-detected from the first row.

4Percent CompleteFloat 0–100 or 0–1 (with ×100 flag)Optional multiplier checkbox converts 0–1 range to 0–100

3 · Dependency / Relationship Types

The predecessor parser (regex-based) supports all four standard PDM relationship types, each optionally qualified with a signed integer lead/lag expressed in working days.

TypeSyntax exampleMeaning
FS (default)3FS-1Finish-to-Start: successor starts after predecessor finishes (− 1 day lead)
SS2SS+2Start-to-Start: successor starts after predecessor starts (+ 2 days lag)
FF4FF+1Finish-to-Finish: successor finishes after predecessor finishes
SF1SFStart-to-Finish: successor finishes after predecessor starts

Multiple predecessors are separated by semicolons. Lead/lag values skip non-working days when the working-day calendar is enabled.

4 · Scheduling Engine — Critical Path Method (CPM)

4.1 Algorithm Steps

  • Topological sort (Kahn's BFS algorithm) to establish a valid processing order.
  • Forward pass — computes Earliest Start Date (ESD) and Earliest Finish Date (EFD) for every task, respecting all four PDM dependency types and lead/lag values, skipping non-working days.
  • Backward pass — computes Latest Start Date (LSD) and Latest Finish Date (LFD) working backwards from project end.
  • Parent task adaptation — parent tasks derive their dates, elapsed, duration, and weighted % complete from their children (deepest-first). The whole cycle repeats until convergence (max 10 iterations) to handle chains of nested parents.
  • Total Float calculation — TF = LSD − ESD in calendar days.
  • Critical path identification — tasks are marked critical when TF = 0 and they lie on the longest path through the project (via recursive path-finding).

4.2 Computed Task Attributes

  • Earliest / Latest Start & Finish Dates
  • Elapsed calendar days (vs. working days)
  • Actual elapsed (capped to today)
  • Ideal percent complete
  • Total float
  • Critical flag
  • Delay flag (actual % < ideal %)
  • Milestone flag (duration = 0)

5 · Working-Day Calendar

A configurable non-working calendar is applied consistently throughout the CPM passes and lead/lag calculations. Settings are stored in window.NonWorkingDatesConfig and persist across JSON save/load cycles.

  • Toggle Saturday and/or Sunday as non-working.
  • Define any number of non-working date periods (from–to ranges).
  • Individual non-working dates are pre-expanded from periods and stored as a sorted array of ISO strings for O(1) daily lookup.
  • Project start date is automatically advanced to the next working day if it falls on a non-working day.

7 · SVG Chart Rendering

7.1 Chart Structure

  • Legend — auto-generated colour key at the top of the SVG.
  • Header timeline — date labels spaced by a configurable minimum pixel gap to prevent overlap; vertical grid lines at every unique task start date.
  • Today line — vertical marker (solid or dashed, configurable colour & width) shown only when today falls within the chart's date range.
  • Horizontal axes — horizontal timeline bar + vertical Y-axis bar, both configurable.
  • Task rows — one horizontal band per task, separated by thin configurable lines.

7.2 Bar Types

TypeShapeDistinguishing behaviour
Normal child task Rounded-corner rectangle Split into "% not complete" and "% complete" layers; optional red critical-path centre stripe
Parent / summary task Rectangle (separate height/space/colour) Same dual-layer split; configurable rounding, height, and spacing differ from children
Milestone Rotated square (rhombus / diamond) Colour reflects completion; date label shown to the right

7.3 Dependency Arrows

SVG <path> elements are drawn for every declared predecessor relationship. The routing algorithm adjusts the elbow path geometry per relationship type (FS, SS, FF, SF) so arrows connect the semantically correct anchor points (end→start, start→start, etc.). An SVG marker (filled arrowhead) is attached to each path end. Lines are independently configurable per task: colour, width, solid/dashed.

7.4 Tooltip Details on Hover

Every bar, milestone, and task label embeds an SVG <title> element containing a full data summary: id, name, duration, elapsed, actual elapsed, all four CPM dates, % complete, ideal %, float, critical/delay/milestone/parent flags, depth, predecessor list, and resources.

8.1 Per-Task Settings Store

A global TasksSettings singleton holds a settings map keyed by task ID. Changes made via the tabbed panels are applied only to the tasks selected in the multi-select Task Selector control (supports Ctrl/Shift multi-selection). This enables per-task visual overrides independent of the global defaults. Settings are serialised into the JSON project file and restored on load.

11 · Interactive Edit Mode

When the Edit Mode checkbox is active, the task-label area of each SVG bar is replaced by a row of inline <foreignObject> input fields rendered inside the SVG. Editing any field triggers immediate CSV synchronisation and chart regeneration.

Editable fields per task row

  • Id (read-only in the form)
  • Name — indented by WBS depth for visual hierarchy
  • Duration — validated ≥ 0
  • % Complete — validated ≥ 0
  • Predecessors — validated against circular-dependency rules before acceptance; rejected IDs shown in alert
  • Resources — free text

Edit Mode toolbar buttons

ButtonFunction
Create TaskInserts a blank task immediately after the selected task; new ID = max existing ID + 1; inherits parent ID from context
Delete TasksDeletes selected tasks plus all their descendants (recursive); cleans orphan predecessor references from remaining tasks
Indent TasksPromotes selected tasks one level deeper by assigning the nearest eligible task above as the new parent; validates post-indent ancestor constraints and removes now-invalid predecessors
Outdent TasksMoves selected tasks one level up (grandparent becomes parent); reorders the CSV block accordingly; removes predecessors that would become invalid after the move

12 · Responsive Scaling & Multi-device Support

A unifiedScale() function calculates a uniform CSS transform: scale() factor based on window.innerWidth / 800 and window.innerHeight / 600, taking the minimum (aspect-ratio-preserving) scale. This is applied to the entire layout wrapper on page load and on every resize event, making the tool usable on tablets and small laptop screens without horizontal scrolling.
